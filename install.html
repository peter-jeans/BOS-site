Cloud BOS install process

Connect an app to Cloud BOS without publishing its secrets.

Cloud BOS should be installed as a governed advisory and update layer. The app keeps its own identity, project memory, and local governance files; Cloud BOS supplies current rules, upgrade plans, scan guidance, and compact capability metadata.

Base44 BC+ derivatives Codex plugin Replit Lovable GitHub + localBOS VS Code-ready path Live-app rescue

Choose the right connector surface

Cloud BOS is the API. Each development environment needs a small local connector or proxy that calls Cloud BOS without exposing secrets or copying private BOS internals.

Codex users

Install the public connector plugin

Use the downloadable cloudbos-public-connector package. It installs into the user's local Codex plugin environment and calls Cloud BOS with that user's local token.

Download Codex connector

App builders

Install a server-side app proxy

Base44, Replit, Lovable, and similar app builders should call Cloud BOS through a backend function or server route. Browser code should not hold Cloud BOS credentials.

Repo workflows

Use localBOS or a thin CLI wrapper

GitHub, VS Code, and terminal workflows should keep project truth in repo-local governance files and use Codex/localBOS or a future thin CLI wrapper for Cloud BOS checks.

The safe install pattern

Every app type uses the same boundary: keep secrets in the app platform's server-side secret store, install only lightweight local governance files, then verify that local BOS is consulted before Cloud BOS fallback.

1

Establish app identity

Record app name, platform, app id where available, owner, environment, lineage, and whether this is a new app, clone, derivative, migration, or live rescue.

2

Install local governance

Add the app's scoped BOS files: app spec, AICONTROL or equivalent instruction file, governance pack, receipts, and capability manifest.

3

Create server-side proxy

The app calls Cloud BOS through a backend function or server route. Browser code should not hold Cloud BOS tokens.

4

Verify local-first routing

Commands such as /update, /upgrade, /scan, and ask BOS should report whether local BOS and Cloud BOS were consulted.

Target app App UI, owner/admin actions, app spec, local governance, receipts.
-->
Server secret store Cloud BOS token or signing secret. Never embed in browser code or public files.
-->
App proxy Allowlisted Cloud BOS routes, request shaping, local file-presence report.
-->
Cloud BOS Current rules, upgrade plans, scan profiles, capability metadata, safe templates.

Install route by app type

Codex plugin

Install locally, then add a user token

  • Download cloudbos-public-connector from this page.
  • Install it into the user's local Codex plugin source or future private marketplace path.
  • Store the user's Cloud BOS token in their local environment, not in the plugin package.
  • Open a new Codex thread and ask: Check Cloud BOS connection status.
  • Expected no-token state is BLOCKED_NO_TOKEN; expected token display is REDACTED.
Base44 direct

Use BuildAI plus a server-side function

  • Install documents/AICONTROL.md or the platform equivalent.
  • Create a backend cloudBosAdvice style proxy with the token stored as a Base44 secret.
  • Install governance/CAPABILITY_MANIFEST.json, governance/BOS_VERSION.json, governance/BOS_STATE.json, and receipts.
  • Run /update and confirm the response reports local BOS first, then Cloud BOS if needed.
BC+ and derivatives

Preserve lineage and scoped project truth

  • Classify the app as parent, clone, derivative, migration target, or rescue target.
  • Keep each app's BOS memory inside that app, not only in masterBOS.
  • Update identity fields when app name, owner, platform id, or environment changes.
  • Use Cloud BOS for upgrade guidance, not as the only holder of app state.
Replit

Use repo files plus environment secrets

  • Install local .bos or bos/ project memory with app spec, manifest, receipts, and routing rules.
  • Store Cloud BOS credentials in Replit Secrets, not in source.
  • Add a server route that forwards only compact metadata and requested Cloud BOS routes.
  • Verify the app can report its environment, app identity, deployment target, and BOS currentness.
Lovable or similar builders

Start with prompt-governance and export checks

  • Install a lightweight instruction layer that names BOS routing, app identity, and no-secrets boundaries.
  • When backend code is available, add the same server-side Cloud BOS proxy pattern.
  • When backend code is not available, use copy-ready BOS prompts and require readback receipts.
  • Before migration or export, trigger the Migration Agent path and record the target runtime.
GitHub + Codex/local repo

Use localBOS as the executor guard

  • Keep source truth in GitHub and local project truth in the repo's scoped BOS files.
  • Use Codex/localBOS for patches, tests, receipts, and verification.
  • Use Cloud BOS for current guidance, upgrades, capability metadata, and safe templates.
  • Never publish .bos, DevPack, private receipts, secrets, or raw logs to the public site.
Live-app rescue

Classify before touching production

  • Identify whether the app has live users, customer records, payments, uploads, or regulated data.
  • Prefer clone, backup, and read-only scans before direct mutation.
  • Record what data exists, where it lives, and which system is the source of truth.
  • Only send sanitized category-level status to Cloud BOS unless an approved data-processing path exists.
VS Code / local terminal

Use the same thin-client contract

  • For now, use Codex/localBOS where available for governed repo work.
  • Do not point VS Code extensions or terminal scripts at masterBOS internals.
  • Future VS Code support should reuse the same Cloud BOS endpoint contract and no-secrets transport rules.
  • Store tokens in the developer's local secret mechanism, never in committed workspace files.

What to configure without exposing it

Use placeholders in public docs and prompts: <CLOUD_BOS_ENDPOINT>, <SERVER_SECRET_NAME>, <APP_ID>, <OWNER_EMAIL>, <ENVIRONMENT>.
Do not paste into public pages or browser code: API tokens, signing secrets, database credentials, customer data, source dumps, raw logs, raw receipts, screenshots with private data, or full private BOS internals.

Allowed install evidence

Installed version, capability ids, file-presence status, app classification, safe route names, and whether local BOS / Cloud BOS were consulted.

Allowed Cloud BOS payload

Compact project metadata, capability requests, currentness checks, sanitized issue category, route id, and consented resolved-pattern summary.

Blocked by default

Source code, app data, user data, secrets, raw prompts, raw logs, raw receipts, private business material, and unredacted customer records.

Smoke test after install

The app should prove connection and routing with safe readback, not by exposing secret values.

Currentness

Run /update or /upgrade. Expected output names installed local version, Cloud BOS route, pending steps, and next action.

Plain language

Ask ask BOS how to improve this page. Expected output says whether local BOS and Cloud BOS were consulted.

No-secret proof

Confirm logs and UI show secret names or statuses only, never token values or private payloads.

Expected readback: LOCAL_BOS_CONSULTED: YES | CLOUD_BOS_CONSULTED: YES/NO | BOS_ROUTE: local-first outcome | CAPABILITY_SPEAKER: BOS-<capability>>

Future public connector package

The public/beta package is separate from the local developer plugin. It is named cloudbos-public-connector so beta users do not confuse it with the local masterBOS development connector.

Download package

The package contains the thin Codex connector only. It does not include a Cloud BOS API key and does not grant access by itself.

Download cloudbos-public-connector

Verify manifest

The manifest lists the package contents, hash, and privacy boundaries for the downloadable zip.

View package manifest
