Cloud BOS onboarding

Install BOS in the app, keep secrets on the server.

The target app keeps its own identity, project memory, and evidence records. Cloud BOS supplies current guidance through a thin connector or server-side proxy.

Start here

Use this sequence before choosing platform-specific details. It is the same whether the app is new, cloned, already broken, or being prepared for beta.

1

Classify the app

Name the app, owner, platform, environment, lineage, stage, and whether this is a new build, clone, derivative, migration, or live rescue.

2

Install local BOS memory

Add an app-scoped instruction layer, app spec, compact governance summary, evidence records, and local-first guidance rules.

3

Add the safe connection

Use a Codex connector, backend function, or server route. Store tokens in the local or server secret store, never in browser code.

4

Run the smoke test

Ask for a BOS update or a plain-language BOS recommendation. The response should say whether local app guidance and Cloud BOS were consulted.

The no-secrets connection boundary

Cloud BOS is the guidance source. The app or developer environment owns the connector and the secret. Public docs, browser code, and downloadable packages must not contain tokens.

Target app App UI, owner/admin actions, app spec, local guidance, evidence records.

-->
App proxy Allowlisted Cloud BOS requests, request shaping, and safe local-currentness status.
-->

Choose the connector for your environment

Most users only need one of these routes. The detail is here so beta testers can install without exposing private app data or confusing the public connector with private BOS internals.

Codex plugin
Ready for beta test

Install locally, then add a user token

  • Download the public connector from this page.
  • Install it into the user's local Codex plugin source or future private marketplace path.
  • Store the user's Cloud BOS token in their local environment, not in the plugin package.
  • Open a new Codex thread and ask for Cloud BOS connection status.
  • Expected output should show a safe connected/not-connected state without exposing token values.
Base44 direct
Ready for beta test

Use BuildAI plus a server-side function

  • Install the platform instruction layer supplied during onboarding.
  • Create a backend proxy with the token stored as a Base44 secret.
  • Add compact local app identity, currentness, and evidence records.
  • Run an update check and confirm the response reports local guidance first, then Cloud BOS if needed.
BC+ and derivatives
Ready for beta test

Preserve lineage and scoped project truth

  • Classify the app as parent, clone, derivative, migration target, or rescue target.
  • Keep each app's BOS memory inside that app, not only in a central development workspace.
  • Update identity fields when app name, owner, platform id, or environment changes.
  • Use Cloud BOS for upgrade guidance, not as the only holder of app state.
Replit
Coming Soon

Use repo files plus environment secrets

  • Install local project memory with app spec, currentness status, evidence records, and routing guidance.
  • Store Cloud BOS credentials in Replit Secrets, not in source.
  • Add a server route that forwards only compact metadata and approved Cloud BOS requests.
  • Verify the app can report its environment, app identity, deployment target, and BOS currentness.
Lovable or similar builders
Coming Soon

  • Install a lightweight instruction layer that names BOS routing, app identity, and no-secrets boundaries.
  • When backend code is available, add the same server-side Cloud BOS proxy pattern.
  • When backend code is not available, use guided BOS prompts and require safe readback evidence.
  • Before migration or export, trigger the Migration Agent path and record the target runtime.
GitHub + Codex/local repo
Ready for beta test

Use localBOS as the executor guard

  • Keep source truth in GitHub and local project truth in scoped project files.
  • Use Codex/localBOS for patches, tests, evidence records, and verification.
  • Use Cloud BOS for current guidance, upgrades, capability metadata, and safe templates.
  • Never publish private project memory, development packs, private records, secrets, or raw logs to the public site.
Live-app rescue
Coming Soon

Classify before touching production

  • Identify whether the app has live users, customer records, payments, uploads, or regulated data.
  • Prefer clone, backup, and read-only scans before direct mutation.
  • Record what data exists, where it lives, and which system is the source of truth.
  • Only send sanitized category-level status to Cloud BOS unless an approved data-processing path exists.
VS Code / local terminal
Coming Soon

Use the same thin-client contract

  • For now, use Codex/localBOS where available for governed repo work.
  • Do not point VS Code extensions or terminal scripts at private BOS internals.
  • Future VS Code support should reuse the same Cloud BOS endpoint contract and no-secrets transport rules.
  • Store tokens in the developer's local secret mechanism, never in committed workspace files.

What to configure without exposing it

Use placeholders in public docs and prompts: <CLOUD_BOS_ENDPOINT>, <SERVER_SECRET_NAME>, <APP_ID>, <OWNER_EMAIL>, <ENVIRONMENT>.
Do not paste into public pages or browser code: API tokens, signing secrets, database credentials, customer data, source dumps, raw logs, raw evidence records, screenshots with private data, or private BOS internals.

Allowed install evidence

Installed version, broad capability area, app classification, safe currentness status, and whether local guidance / Cloud BOS were consulted.

Allowed Cloud BOS payload

Compact project metadata, broad capability requests, currentness checks, sanitized issue category, and consented resolved-pattern summary.

Blocked by default

Source code, app data, user data, secrets, raw prompts, raw logs, raw evidence records, private business material, and unredacted customer records.

Smoke test after install

The app should prove connection and routing with safe readback, not by exposing secret values.

Currentness

Ask for an update or upgrade check. Expected output names installed local version, broad Cloud BOS consultation status, pending steps, and next action.

Plain language

Ask ask BOS how to improve this page. Expected output says whether local BOS and Cloud BOS were consulted.

No-secret proof

Confirm logs and UI show secret names or statuses only, never token values or private payloads.

Expected readback: local guidance consulted, Cloud BOS consulted when needed, no secrets exposed, next action clear.
